# AI Light Novel Translator (Ollama Pipeline)

Локальный конвейер перевода корейских ранобэ на русский язык. Работает на
[Ollama](https://ollama.ai/) с несколькими моделями: одна переводит, другая
независимо проверяет, третья полирует, четвёртая судит. Опционально может
вызывать облачный Gemini, чтобы добить сегменты, которые не вытянули локальные
модели.

## Как это работает

Пайплайн разбит на девять стадий. Грубо: текст режется на сегменты, из них
извлекается глоссарий имён и терминов, две модели независимо его сверяют,
дальше идёт построчный перевод с опорой на этот глоссарий, потом снова
билингвальная проверка, литературная полировка, мехQA, оценка качества
LLM-судьёй, и финальная починка проблемных мест.

Идея в слоях: каждая стадия ловит свой класс ошибок. **qwen** хорошо переводит
смысл, но иногда галлюцинирует или калькирует; **aya** другой семьи и ловит
именно это; **gemma** делает русский живым, не трогая смысл; **magistral**
оценивает результат как независимый редактор; **Gemini** добивает то, что
осталось проблемным.

Запускать всё одной командой через `run_all.bat`. Каждая стадия сохраняет
прогресс посегментно и поддерживает resume — можно остановить bat в любой
момент (`Ctrl+C`) и запустить снова, продолжит с того места, где остановился.

## Установка

```bash
pip install -r requirements.txt
ollama serve
ollama pull qwen3:30b-a3b-instruct-2507-q4_K_M
ollama pull aya-expanse:32b
ollama pull gemma3:27b
ollama pull magistral:24b
```

`google-genai` из `requirements.txt` нужен только для Gemini-фолбэка. Если
облако не планируется — можно не ставить (`pip install -q -U google-genai`
поставит отдельно потом).

Корейский исходник положи в `input/source.txt`. Ключ от Gemini API (если
есть) — в `apikey.txt` рядом со скриптами (или env-переменная `GEMINI_API_KEY`).

## Запуск

Просто двойной клик по `run_all.bat`. Он по очереди прогоняет все стадии,
пишет полный лог в `output/pipeline.log`, проверяет, что Ollama жива.

Скрипт **resume-friendly**: если остановил, запусти заново — стадии 1–3
пропустятся по своим done-маркерам, стадии 4–9 продолжат с того сегмента,
на котором остановились.

> Для НОВОЙ главы перед запуском очисти или перемести папку `output/` —
> иначе пайплайн переиспользует старые сегменты, глоссарий и переводы.

Или те же стадии вручную:

```bash
python stage1_split.py
python stage2_glossary.py
python stage3_glossary_check.py --resume
python gemini_fallback.py --mode glossary
python stage4_translate.py --resume
python stage5_translate_check.py --resume
python gemini_fallback.py --include-check-failed
python stage6_proofread.py --resume --export-txt
python gemini_fallback.py --mode polish
python stage7_qa.py
python stage8_judge.py --resume
python stage9_judge_fix.py --resume
python export_final.py --format plain
```

## Стадии пайплайна

**Стадия 1 — нарезка.** `stage1_split.py` режет `input/source.txt` на сегменты
размером до 1500 символов по границам предложений (regex по корейским
окончаниям, опционально `--method kss` для более точной разбивки через `kss+pecab`).
Выход: `output/1_segments.json`.

**Стадия 2 — извлечение глоссария.** `stage2_glossary.py` (`qwen`) читает
текст батчами по 5 сегментов и вытаскивает имена/термины/места/титулы/навыки.
Для сино-корейских концепт-терминов восстанавливает ханча (`구음절맥` →
`九陰絶脈`) — это потом подсказка переводчику, что переводить нужно по смыслу
иероглифов, а не фонетикой. После сбора делает второй проход «нормализации»
(чанками по 40, та же qwen), чтобы убрать дубли и привести разные написания
одного имени к единому виду. В конце принудительно сеет фиксированный список
обращений из `config.HONORIFICS` (`도련님` → «молодой господин» и т.п.).
Выход: `output/2_glossary.json` + чекпойнт `2_glossary.json.checkpoint.jsonl`.

**Стадия 3 — кросс-проверка глоссария.** `stage3_glossary_check.py` (`aya-expanse`)
независимо проверяет глоссарий чанками по 40 терминов. Меняет ТОЛЬКО русскую
сторону (`korean` и `hanja` неприкосновенны), может править фонетические
кальки концепт-терминов, заимствования, неверные титулы, ошибки гендера.
Сохраняет прогресс посегментно. По завершении пишет done-маркер
`2_glossary.checked.json`, список расхождений с qwen в
`2_glossary_disagreements.txt` и `.json`. Пристный qwen-вариант сохраняется
в `2_glossary.qwen.json` (на случай переразбора).

**Стадия 4 — арбитр глоссария (Gemini).** `gemini_fallback.py --mode glossary`
берёт `2_glossary_disagreements.json` и по каждому спорному термину спрашивает
Gemini «qwen или aya, или предложи третий вариант». Применяет вердикт к
`2_glossary.json`, помечает термины полем `arbiter=gemini`. Идемпотентно
пропускает уже размеченных при повторном запуске. Без ключа Gemini шаг
просто пропускается, пайплайн продолжает с qwen+aya глоссарием. Лог:
`2_glossary_gemini_arbiter.txt`.

**Стадия 5 — перевод.** `stage4_translate.py` (`qwen`) переводит каждый сегмент,
показывая модели в промпте только релевантные для этого сегмента термины из
глоссария (substring-матч по корейскому, с лимитом 60 терминов на промпт).
Использует бегущее «саммари по истории» (story bible) для связности между
сегментами — без сырого предыдущего перевода, чтобы избежать галлюцинаций-копий.
Каждый сегмент сохраняется атомарно. Выход: `output/3_translated.json`.

**Стадия 6 — кросс-проверка перевода.** `stage5_translate_check.py` (`aya-expanse`)
билингвально сверяет каждый перевод с корейским оригиналом. В отличие от
stage3, тут aya видит и оригинал, и перевод вместе. Правит ТОЛЬКО смысловые
искажения, не трогает стилистику. Особое внимание выравниванию реплик: если
в русском есть строки, которых нет в корейском (галлюцинации-копии) — удаляет.
Сохраняет в тот же `3_translated.json`, исходный qwen-вариант — в
`3_translated.qwen.json`.

**Стадия 7 — Gemini-фолбэк для проблемных переводов.**
`gemini_fallback.py --include-check-failed` ищет сегменты с ошибками
(остатки корейского, мало кириллицы, петли повторов, `[ОШИБКА ПЕРЕВОДА]`,
не прошедшие проверку aya) и переводит их облачным Gemini. Помечает
`translated_by="gemini"`, прежняя попытка сохраняется в `failed_translation`.
Без ключа Gemini шаг пропускается.

**Стадия 8 — литературная вычитка.** `stage6_proofread.py` (`gemma3`) проходит
по русскому тексту и делает его живым: убирает кальки, чинит грамматику,
улучшает ритм. У gemma НЕТ корейского оригинала под рукой — она редактор
русского, не переводчик, и не должна менять смысл. Длина результата
ограничена коридором ±20–25% от черновика, иначе откатываемся на черновик
(`fallback_reason`). Выход: `output/4_final.json` (+ `4_final.txt` если указан
`--export-txt`).

**Стадия 9 — Gemini полировка fallback-сегментов.**
`gemini_fallback.py --mode polish` чистит сегменты, где gemma не справилась
(есть `fallback_reason`). Меняет только форму, не смысл.

**Стадия 10 — мехQA.** `stage7_qa.py` — без обращения к моделям. Считает
доли корейских символов в русском, ловит «утерянные» термины глоссария
(если корейский ключ был в исходнике, а его русский эквивалент не нашёлся
в переводе), сильные перекосы длин, дубли абзацев внутри сегмента. Выход:
`output/5_qa_report.json` + `5_qa_report.txt`.

**Стадия 11 — финальный судья.** `stage8_judge.py` (`magistral`) для каждого
сегмента билингвально выставляет баллы (accuracy / fluency / terminology /
completeness, 1–5) и выдаёт конкретный список проблем (severity, type, quote,
comment) + verdict (`ok / review / bad`). НИЧЕГО НЕ МЕНЯЕТ в переводе.
Magistral — рассуждающая модель, её `<think>` блоки скрипт срезает сам.
Выход: `output/6_judge_report.json` + `6_judge_report.txt` (отсортирован по
серьёзности).

**Стадия 12 — починка по вердикту судьи (Gemini).** `stage9_judge_fix.py`
для каждого сегмента с `verdict=bad` или `verdict=review` с `severity=high`
шлёт в Gemini корейский оригинал + текущий русский + список конкретных
замечаний судьи, и просит починить ровно эти места. Есть cost-gate (по
умолчанию <100 запросов автоматически, иначе требует `--yes`). Resume по
полю `judge_fix_applied`. Лог: `6_judge_fixes.txt`.

**Финал — экспорт.** `export_final.py --format plain` собирает `4_final.json`
в чистый русский текст `output/final_text.txt`. Доступны форматы:
`plain` (только русский), `with_source` (KO + RU абзацами), `side_by_side`
(для сравнения).

## Что лежит в `output/`

| Файл | Что это |
| --- | --- |
| `1_segments.json` | Нарезанный корейский текст |
| `2_glossary.json` | Финальный глоссарий (после всех проверок и арбитра) |
| `2_glossary.qwen.json` | Пристный qwen-вариант глоссария (бэкап) |
| `2_glossary.checked.json` | Done-маркер stage3 (не редактировать) |
| `2_glossary_disagreements.txt` / `.json` | Где qwen и aya разошлись |
| `2_glossary_gemini_arbiter.txt` | Что выбрал Gemini-арбитр и почему |
| `2_glossary_overrides.json` | Ручные оверрайды (создаёшь сам, высший приоритет) |
| `3_translated.json` | Перевод сегментов (после aya-проверки + gemini fallback) |
| `3_translated.qwen.json` | Сырой qwen-перевод до aya (бэкап) |
| `4_final.json` | Финальный JSON со всеми правками (включая stage9) |
| `4_final.txt` / `final_text.txt` | Чистый русский текст для чтения |
| `5_qa_report.txt` | Мех-QA отчёт (где что подозрительно) |
| `6_judge_report.txt` | Отчёт судьи (отсортирован по серьёзности проблем) |
| `6_judge_fixes.txt` | Лог правок Gemini по вердикту судьи |
| `pipeline.log` | Полный лог прогона bat |
| `backups/` | Rolling-бэкапы стадий (`*.backup_YYYYMMDD_HHMMSS.json`) |

Для чтения — `final_text.txt`. Если в нём попадаются `[ОШИБКА ПЕРЕВОДА]` —
смотри `6_judge_report.txt`, там будет указан сегмент, и можно вручную
добить через `python gemini_fallback.py --ids N --also-final`.

## Облачный фолбэк через Gemini

Gemini в этом пайплайне — это **необязательный** четырёхрежимный помощник,
который добивает то, с чем локальные модели не справились. Все четыре места
работают без внимания: подвязаны в `run_all.bat`, если ключа нет — просто
пропускаются, пайплайн идёт дальше.

```bash
# 1. Арбитр спорных терминов глоссария (после stage3)
python gemini_fallback.py --mode glossary

# 2. Перевод сегментов, которые qwen+aya не вытянули (после stage5)
python gemini_fallback.py --include-check-failed
python gemini_fallback.py --ids 6,25                  # вручную, конкретные id
python gemini_fallback.py --ids 6 --also-final        # ещё и обновить 4_final.json

# 3. Полировка сегментов, которые gemma вынуждена была откатить (после stage6)
python gemini_fallback.py --mode polish
python gemini_fallback.py --mode polish --ids 14      # вручную

# 4. Починка по вердикту судьи (после stage8)
python stage9_judge_fix.py --resume                   # с cost-gate (<100 авто)
python stage9_judge_fix.py --resume --yes             # без подтверждения
python stage9_judge_fix.py --resume --limit 50        # порциями
python stage9_judge_fix.py --ids 47,113               # вручную
```

Ключ хранится в `apikey.txt` или env `GEMINI_API_KEY` / `GOOGLE_API_KEY`.
Модель: `MODEL_CLOUD` в `config.py` (по умолчанию `gemini-3.5-flash`).

**Cost gate stage9.** Перед отправкой считает кандидатов. Если их меньше 100
(порог `--max-auto`) — отправляет сразу. Если больше — в интерактивном
терминале спрашивает подтверждение, а в режиме под bat (вывод в лог)
выходит с кодом 1 и просит запустить вручную с `--yes`. bat это переживёт.

**Quota auto-abort.** Если Gemini в 3 запросах подряд возвращает пустой ответ
(вероятно исчерпан лимит) — stage9 останавливается, чтобы не молотить попусту.
На следующий день `--resume` продолжит с того же сегмента.

## Ключевые настройки (`config.py`)

| Параметр | Что регулирует |
| --- | --- |
| `MODEL_TRANSLATE`, `MODEL_CHECK`, `MODEL_PROOFREAD`, `MODEL_JUDGE` | Локальные модели Ollama |
| `MODEL_CLOUD`, `APIKEY_FILE` | Облачный Gemini (фолбэк, арбитр, судья-фикс) |
| `HONORIFICS` | Фиксированные обращения, всегда в глоссарии |
| `TRANSLATE_USE_PREV_CONTEXT` | Сырой контекст пред. сегмента в промпте (по умолч. `False`) |
| `TRANSLATE_TOP_P` / `TOP_K` / `*_PENALTY` | Декодирование переводчика |
| `SUMMARY_UPDATE_EVERY` | Частота обновления бегущего саммари (`0` — выкл) |
| `GLOSSARY_NORMALIZE_CHUNK` | Размер чанка при нормализации глоссария и stage3 |
| `GLOSSARY_OVERRIDES_FILE` | Путь к ручным оверрайдам глоссария |
| `QA_*` | Пороги срабатывания флагов на этапе мехQA |
| `JUDGE_*` | Контекст / температура / лимит предикта судьи |
| `BACKUP_DIR` | Папка для rolling-бэкапов (`output/backups/`) |
