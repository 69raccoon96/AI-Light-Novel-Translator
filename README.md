# AI Light Novel Translator (Ollama Pipeline)

Автоматизированный многоагентный конвейер (пайплайн) для перевода корейских ранобэ и веб-новелл на русский язык. Проект построен на базе локального сервера [Ollama](https://ollama.ai/) и использует архитектуру из нескольких узкоспециализированных LLM для достижения максимального качества перевода.

Главная архитектурная особенность — **строгое разделение ролей**. Каждая стадия перевода выполняется отдельной моделью, которая проверяет, корректирует или оценивает работу предыдущей. Опционально реализован облачный фолбэк (Google Gemini) для обработки сложных сегментов и внесения точечных правок.

## 🧠 Распределение ролей (Модели по умолчанию)

*   **Переводчик (`qwen3:30b...`)**: Извлекает глоссарий, переводит текст с учетом восстановленных иероглифов (ханча) и поддерживает общую связность.
*   **Контролер (`aya-expanse:32b`)**: Билингвально сверяет черновой перевод с оригиналом. Жестко пресекает галлюцинации (выдуманные реплики) и смысловые искажения.
*   **Редактор (`gemma3:27b`)**: Выполняет литературную полировку русского текста, не имея доступа к оригиналу (гарантия сохранения смысла от предыдущих этапов).
*   **Судья (`magistral:24b`)**: LLM-as-a-Judge. Оценивает финальный текст по критериям точности, терминологии и стилистики, формируя список конкретных замечаний.
*   **Ассистент-хирург (Облако `gemini-3.5-flash`)**: Разрешает споры локальных моделей, переводит сленг/системные логи и применяет правки судьи.

## 🚀 Установка и настройка

1. **Клонируйте репозиторий и установите зависимости:**
   ```bash
   pip install -r requirements.txt
   ```
2. **Установите [Ollama](https://ollama.ai/) и загрузите модели:**
   ```bash
   ollama pull qwen3:30b-a3b-instruct-2507-q4_K_M
   ollama pull aya-expanse:32b
   ollama pull gemma3:27b
   ollama pull magistral:24b
   ```
3. **Настройте API ключ Gemini (Опционально, но рекомендуется):**
   Создайте файл `apikey.txt` в корневой папке проекта и поместите туда ключ Google Gemini API, либо задайте переменную окружения `GEMINI_API_KEY`.

## 📖 Использование (Повседневный рабочий процесс)

Пайплайн спроектирован с учетом **устойчивости к прерываниям (idempotency)**. Прогресс сохраняется по каждому сегменту текста. При остановке скрипта (например, `Ctrl+C`) повторный запуск продолжит работу с места остановки.

**Алгоритм перевода новой главы:**
1. Очистите папку `output/` (в противном случае пайплайн продолжит работу с предыдущим текстом и глоссарием).
2. Поместите исходный корейский текст в файл `input/source.txt`.
3. Запустите командный файл:
   ```cmd
   run_all.bat
   ```
4. По завершении всех стадий заберите готовый текст из файла `output/final_text.txt`.

## ⚙️ Архитектура конвейера (13 этапов)

Выполнение `run_all.bat` разбито на 4 логические фазы:

### Фаза 1: Подготовка и Глоссарий
*   **Шаг 1 (Split):** Разделение исходного текста на сегменты по границам предложений (регулярные выражения или библиотека `kss`).
*   **Шаг 2 (Glossary):** Qwen извлекает имена, титулы и термины. Концепт-терминам восстанавливается ханча для перевода по смыслу иероглифов. Выполняется дедупликация и внедрение гоноративов (обращений).
*   **Шаг 3 (Glossary Check):** Aya независимо проверяет русскую часть глоссария на ошибки транслитерации и терминологии.
*   **Шаг 4 (Gemini Arbiter):** При наличии расхождений между Qwen и Aya, Gemini выбирает правильный вариант перевода.

### Фаза 2: Черновой перевод
*   **Шаг 5 (Translate):** Qwen переводит сегменты, используя релевантную часть глоссария. Для поддержания контекста используется механизм "бегущего саммари" (Story Bible).
*   **Шаг 6 (Translate Check):** Aya сверяет перевод с оригиналом. Удаляет лишние (сгаллюцинированные) предложения и восстанавливает выравнивание реплик.
*   **Шаг 7 (Fallback):** Gemini переводит сегменты, вызвавшие ошибки у локальных моделей (например, обилие корейского сленга, неформатные символы).

### Фаза 3: Литературная обработка
*   **Шаг 8 (Proofread):** Gemma полирует текст, устраняя канцелярщины и машинные кальки. Действует система жестких лимитов: при сильном изменении объема текста происходит откат к черновому переводу.
*   **Шаг 9 (Polish Fallback):** Gemini исправляет сегменты, отбракованные на предыдущем шаге (возвращенные к черновому варианту).

### Фаза 4: Контроль качества и Финал
*   **Шаг 10 (QA):** Скриптовая (алгоритмическая) проверка на остатки корейских символов, потерю терминов глоссария и аномалии длины.
*   **Шаг 11 (Judge):** Magistral (LLM-as-a-Judge) оценивает перевод и генерирует отчет с конкретными замечаниями (accuracy, fluency, completeness).
*   **Шаг 12 (Judge Fix):** Gemini применяет исправления к тексту строго по замечаниям Судьи. Работает Cost Gate (защита бюджета API).
*   **Шаг 13 (Export):** Сборка финального файла `final_text.txt`.

## 📂 Структура выходных файлов (`output/`)

| Файл | Назначение |
| :--- | :--- |
| `1_segments.json` | Массив нарезанных сегментов исходного текста. |
| `2_glossary.json` | Итоговый глоссарий после всех проверок и арбитража. |
| `2_glossary_overrides.json` | Файл для ручного переопределения терминов глоссария (высший приоритет). Создается пользователем. |
| `3_translated.json` | Черновой перевод (до литературной редактуры). |
| `4_final.json` | Финальный перевод со всеми метаданными и историей правок. |
| **`final_text.txt`** | **Чистый русский текст (итоговый результат).** |
| `5_qa_report.txt` | Отчет механической проверки (потерянные слова, остатки хангыля). |
| `6_judge_report.txt` | Развернутый аудит качества от модели-судьи. |
| `pipeline.log` | Подробный журнал работы `run_all.bat`. |
| `backups/` | Автоматические резервные копии JSON-файлов на каждом этапе. |

## 🛠 Конфигурация (`config.py`)

Пайплайн обладает высокой степенью гибкости. Все основные параметры вынесены в `config.py`.

### 1. Настройка моделей и API
| Параметр | Описание |
| :--- | :--- |
| `MODEL_GLOSSARY`, `MODEL_TRANSLATE` | Локальные модели для извлечения терминов и базового перевода (Qwen). |
| `MODEL_CHECK` | Локальная модель для кросс-проверки глоссария и перевода (Aya). |
| `MODEL_PROOFREAD` | Локальная модель для литературной редактуры (Gemma). |
| `MODEL_JUDGE` | Модель-судья с поддержкой рассуждения (`<think>`) (Magistral). |
| `MODEL_CLOUD`, `APIKEY_FILE` | Настройки облачного фолбэка (Gemini). |

### 2. Параметры генерации и декодирования
| Параметр | Описание |
| :--- | :--- |
| `*_TEMPERATURE` | Температуры для разных стадий (по умолчанию 0.2 - 0.3 для стабильности). |
| `*_NUM_CTX` | Размер контекстного окна для каждой стадии (важно для предотвращения обрезания длинных сегментов). |
| `TRANSLATE_TOP_P`, `TRANSLATE_TOP_K` | Параметры семплирования при переводе. |
| `TRANSLATE_REPEAT_PENALTY` | Штраф за повторения токенов (помогает от "залипания" модели). |

### 3. Логика перевода и контекст
| Параметр | Описание |
| :--- | :--- |
| `TRANSLATE_USE_PREV_CONTEXT` | Подача предыдущего абзаца в промпт (по умолч. `False`, т.к. провоцирует галлюцинации). |
| `SUMMARY_UPDATE_EVERY` | Частота обновления бегущего саммари (Story Bible). `0` — отключить. |
| `SUMMARY_MAX_CHARS` | Максимальный размер саммари, передаваемого в промпт. |

### 4. Обработка текста и Глоссарий
| Параметр | Описание |
| :--- | :--- |
| `MAX_SEGMENT_CHARS` | Максимальная длина сегмента (по умолч. 1500). |
| `HONORIFICS` | Жестко заданный словарь гоноративов и обращений (внедряется в глоссарий для защиты от фонетической транслитерации). |
| `GLOSSARY_NORMALIZE_CHUNK` | Размер батча терминов при дедупликации глоссария. |

### 5. Пороги контроля качества (QA)
| Параметр | Описание |
| :--- | :--- |
| `QA_KOREAN_RATIO_WARN` | Доля корейских символов, вызывающая срабатывание триггера (по умолч. 3%). |
| `QA_LEN_RATIO_LOW/HIGH` | Допустимый коэффициент изменения длины текста при переводе. |

## 🔒 Защита бюджета API (Cost Gate)

Скрипт `stage9_judge_fix.py` использует платное/лимитированное API. Чтобы предотвратить неконтролируемые расходы, предусмотрен механизм защиты:
* Параметр `--max-auto` (по умолчанию 100) задает лимит запросов.
* Если проблемных сегментов больше, выполнение в автоматическом режиме прервется с ошибкой в логах.
* Для обхода потребуется ручной запуск с флагом `--yes` или настройка `--limit`.
* Также реализован **Auto-abort**: скрипт остановится, если получит 3 пустых ответа подряд от Gemini (признак исчерпания квот). Оставшиеся сегменты будут обработаны при следующем запуске пайплайна (сработает система Resume).
